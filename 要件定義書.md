# UML自動生成ツール 要件定義書

## 1. 文書概要

### 1.1 文書目的
本書は、複数のプログラミング言語を静的解析し、UML および関連図を自動生成するツールの要件を定義することを目的とする。

### 1.2 前提
- 本資料は現時点では叩き台ではなく、**初版実装に着手できる粒度まで具体化した要件定義**として扱う
- ツールの**実装言語は Python のみ**とする
- ツールの**初版解析対象言語は Python のみ**とする
- ツールの提供形態は**ローカル GUI アプリケーション**とし、Web 版は初版・将来対応ともにスコープ外とする
- 将来対応候補として以下の解析言語を見据える
  - TypeScript
  - Java
  - C#
  - JavaScript
  - C
- 将来対応候補を追加する場合も、**初版で確定した中間モデルの骨格は変更しない**ことを原則とする
- Python 初版では以下を主な対応構文とする
  - `class`
  - `def`
  - `import`
  - 継承
  - 属性代入
  - `if`
  - `for`
  - `while`
  - `return`
  - 関数呼び出し
  - クラス生成
  - `@property`
  - `@classmethod`
  - `@staticmethod`
- Python 初版では以下は対象外とする
  - list 内包表記
  - dict / set 内包表記
  - 三項演算子
  - `async` / `await` の詳細追跡
  - generator / `yield` の詳細追跡

### 1.3 開発方針
- 言語ごとの差異を吸収するため、**単一の共通中間モデル**を導入する
- 各言語ごとに Parser / Resolver / Adapter を実装し、共通中間モデルへ変換する
- 図生成処理は可能な限り共通化する
- 完全な情報が取得できない場合も、欠損を許容しつつ出力可能とする
- 動的型付け言語については、必要に応じて**起点関数ベースの型推論**により精度を向上させる
- 将来の対応言語・図種・出力記法が増えても、**中間モデルの破壊的変更を行わずに拡張できる構造**を初版で確定する

---

## 2. システム概要

### 2.1 システム目的
対象ソースコードを静的解析し、以下の図を自動生成する。

#### 初版対応
- クラス図
- モジュール依存図
- シーケンス図

#### 将来対応
- コンポーネント図
- アクティビティ図
- 状態遷移図
- 通信図
- その他、コード解析結果から有用に生成可能な図

### 2.2 スコープ整理
主要スコープは以下の通りとする。

| 領域 | 初版対応 | 将来対応 |
|------|----------|----------|
| 解析対象言語 | Python | TypeScript / Java / C# / JavaScript / C |
| 出力図 | クラス図 / モジュール依存図 / シーケンス図 | コンポーネント図 / アクティビティ図 / 状態遷移図 / 通信図 |
| 出力記法 | PlantUML | Mermaid / Graphviz DOT / XMI / その他 |
| 外部ライブラリ表現 | 非表示 | note 表示、participant + call / return 表示 |
| participant 単位 | クラス単位 | インスタンス単位 |
| 起点関数数 | 1 件 | 複数件 |
| Python 高度構文 | 一部対象外あり | list 内包表記 / 三項演算子 / async / await / yield 等の拡張 |
| 提供形態 | ローカル GUI アプリ | Web 版は非対応 |
| 実行時指定項目 | GUI で都度指定 | 対象項目追加 |
| 固定設定管理 | `config.ini` | 項目拡張 |
| 出力物 | GUI プレビュー + `.puml` 保存 | PNG / SVG 等の画像出力 |

### 2.3 操作形態
- 本ツールは**ローカル GUI アプリケーション**として提供する
- ユーザが都度変更する項目は、**GUI から指定**する
  - 解析対象ディレクトリ
  - 出力先
  - 出力対象図
  - 起点関数
  - 型推論実施有無
- 実行時に毎回変更しない固定寄りの設定は、**`config.ini`** に保持する
- 初版では **GUI 上で PlantUML をプレビュー可能**とし、正本は **`.puml` ファイル保存**とする
- PNG / SVG 等の画像出力は将来対応とする

### 2.4 アーキテクチャ概要
システム構成は以下の層を基本とする。

1. GUI Layer
2. File Collection Layer
3. Parser Layer
4. Symbol Collection Layer
5. Symbol Resolution Layer
6. Intermediate Model Generation Layer
7. Type Inference Layer
8. Diagram Generator Layer
9. Preview / Export Layer

```text
ユーザ操作（ローカル GUI）
    ↓
File Collection
    ↓
Parser Layer（言語依存）
    ↓
Symbol Collection
    ↓
Symbol Resolution
    ↓
Intermediate Model Generation
    ↓
Type Inference（必要時）
    ↓
Diagram Generator
    ↓
GUI Preview / .puml Export
```

### 2.5 基本設計思想
- AST をそのまま共通化しない
- 言語構文ではなく、**意味単位**で中間モデル化する
- 中間モデルは **宣言情報 + 型参照 + 参照情報 + 推論事実 + 振る舞い IR** を中心とした構造とする
- **宣言系データを一次情報（正本）**とし、`Relation` は図生成・探索・検索のための**派生ビュー / 索引情報**として扱う
- **Symbol は共通ヘッダ、各種 Decl / Payload は詳細本体**として責務分離する
- 不明な情報は破棄せず、Unknown / unresolved / confidence / raw などで保持する
- **シンボル解決用の完全修飾名**と**図出力用の表示名**は分離して管理する
- 図生成では AST ではなく中間モデルのみを参照する
- シーケンス図生成や分岐追跡のため、線形ログではなく **Behavior IR** を一次情報として保持する
- 将来対応項目を追加する場合は、CoreModel / Semantic Payload Layer / Behavior IR Layer / Inference Layer の**既存契約を維持したまま拡張**する

## 3. 対象言語要件

### 3.1 初版正式対応言語
初版で正式対応する解析言語は **Python のみ** とする。

### 3.2 将来対応言語
将来対応候補として以下を記載する。

| 言語 | 想定用途 | 特徴 | 将来対応方針 |
|------|----------|------|--------------|
| TypeScript | クラス・interface・型解析 | 型情報が豊富 | 優先対応候補 |
| Java | クラス図生成 | 継承・interface が明確 | 優先対応候補 |
| C# | クラス図生成 | property・namespace がある | 優先対応候補 |
| JavaScript | クラス・関数解析 | 型情報欠損が多い | Unknown を前提に段階対応 |
| C | struct・関数・include 解析 | OO ではない | Procedural Profile を用いて対応 |
| Python | 初版対応済 | 動的型付け、型ヒントは任意 | 初版基準言語 |

### 3.3 初版 Python 対応方針
- 型不明を許容し、必要に応じて起点関数ベースの型推論を行う
- `self.xxx` / `cls.xxx` / import / alias / scope を考慮してシンボル解決する
- 分岐・ループを Behavior IR に保持し、クラス図・依存図・シーケンス図に再利用する

### 3.4 将来対応時の原則
- 将来の各言語は、**言語パーサ・言語別 resolver・言語別 adapter の追加**で吸収する
- 中間モデル自体を言語ごとに作り直さない
- 中間モデルの破壊的変更を伴う将来対応は原則禁止とする

---

## 4. 中間モデル方針

### 4.1 分割方針
中間モデルは、言語系統ごとに別モデルを持つのではなく、以下の構成とする。

- CoreModel
- Semantic Payload Layer
- Behavior IR Layer
- Language Extension
- Inference Layer
- Analysis Support Layer

### 4.2 分割しない理由
対象言語間には差異があるものの、図生成に必要な主要情報は共通している。

- ファイル / モジュール
- 型宣言
- 関数 / メソッド
- フィールド / 変数
- 依存関係
- 呼び出し関係
- 推論された型事実
- 分岐やループを含む振る舞い情報

したがって、初版段階で中間モデルを統一し、Payload 層・Behavior IR 層・Extension 層・Inference 層・Analysis Support 層で吸収する方が保守性・拡張性が高い。

### 4.3 初版で中間モデルを確定する方針
- 初版時点で、**将来対応言語・将来対応図種・将来対応出力記法を見据えた中間モデルの骨格を確定**する
- 将来対応では、既存レイヤに対する **拡張情報の追加** は許容する
- 将来対応では、既存レイヤの **責務変更・ID体系変更・一次情報と派生情報の逆転** は行わない
- 特に以下は初版で固定する
  - `Symbol` を共通ヘッダとする構造
  - `Decl / Payload` を詳細本体とする構造
  - `Relation` を派生ビューとして扱う方針
  - `TypeRef` の正規化方針
  - `Behavior IR` を一次情報とする方針
  - `TypeFact` を上書きではなく蓄積として扱う方針

### 4.4 将来の拡張方針
将来的に差異が大きくなった場合は、別中間モデルではなく、まず以下のような**プロファイル分離**を検討する。

- OOProfile
- ProceduralProfile
- DynamicTypingProfile

## 5. 出力記法要件

### 5.1 初版正式対応
初版の出力記法は **PlantUML のみ** とする。

### 5.2 初版で PlantUML のみとする理由
- クラス図・シーケンス図の表現力が高い
- 初版で最も重視する図種との親和性が高い
- 記法差分の吸収コストを抑え、解析精度と中間モデル品質の確保に集中できる

### 5.3 将来対応
将来対応候補として以下を記載する。

- Mermaid
- Graphviz DOT
- XMI
- D2
- nomnoml
- yUML
- Structurizr DSL

### 5.4 将来対応時の原則
- 新たな出力記法を追加する場合でも、中間モデルおよび図生成インターフェースの大幅改修を必要としない構造とする
- 出力記法追加は **Generator の追加** で吸収することを原則とする

## 7. 解析フロー要件

### 7.1 全体フロー
解析処理は以下の順序で実行する。

1. GUI から解析対象ディレクトリ・出力先・対象図などを受付
2. 対象ファイル収集
3. AST 生成
4. シンボル収集
5. シンボル解決
6. 型未確定を許容した中間モデル生成
7. 必要に応じて GUI に起点関数候補一覧を提示
8. 必要に応じてユーザが GUI 上で起点関数を 1 件選択
9. 必要に応じて起点関数から型推論・呼び出し追跡を実施
10. PlantUML 生成
11. GUI プレビュー表示および `.puml` 保存

### 7.2 解析対象ディレクトリ
- ユーザは **GUI 上で**解析対象の親ディレクトリを指定する
- 当該ディレクトリ配下の Python ファイルを解析対象とする
- 除外対象ディレクトリやファイルの指定は将来対応とする
- 出力先ディレクトリ、生成対象図、型推論実施有無などの実行時項目も GUI 上で指定する
- `config.ini` は固定寄りの設定値保持に利用し、実行ごとに変更する値の主入力手段とはしない

### 7.3 層ごとの責務
#### GUI Layer
- 解析対象ディレクトリ・出力先・出力対象図・起点関数などの入力受付
- 起点関数候補一覧の表示と選択受付
- 生成済み PlantUML のプレビュー表示
- `.puml` 保存操作の受付

#### Parser Layer
- Python ファイルの AST 生成
- 宣言候補・参照候補・構文種別の抽出
- 振る舞い IR 生成に必要な基本構文情報の抽出
- raw AST 由来情報の拡張領域への受け渡し

#### Symbol Resolution Layer
- import 解決
- alias 解決
- scope 解決
- qualified name 解決
- `self.xxx` / `cls.xxx` の解決
- 相対 import の解決
- 解決候補の信頼度付け

#### Intermediate Model Generation Layer
- 中間モデルへの正規化
- Symbol / Payload / TypeRef / Behavior IR の構築
- Diagnostics の記録

#### Type Inference Layer
- 起点関数を起点とした型推論
- 分岐・ループを考慮した型事実蓄積
- シーケンス図生成向けの追跡結果生成

## 8. シンボル解決と型推論

### 8.1 基本方針
**シンボル解決**と**型推論**は別工程として扱う。

- **シンボル解決**: 参照名がどの定義を指すかを決定する工程
- **型推論**: 変数・属性・引数・戻り値・呼び出し対象の型を決定する工程

### 8.2 分離する理由
- 責務分離により実装を明確化できる
- エラー原因の切り分けが容易になる
- 型推論ロジックを中間モデル上で共通化しやすくなる
- 将来の多言語対応時に拡張しやすい

### 8.3 初版における責務整理
初版では Python のみを対象とし、以下の責務分担とする。

- **Parser Layer**  
  Python 構文を認識し、宣言候補・参照候補・制御構造候補を抽出する
- **Symbol Resolution Layer**  
  抽出済み参照がどのシンボルを指すかを解決する
- **Type Inference Layer**  
  解決済み参照と Behavior IR を基に型を推論する

### 8.4 シンボル解決の対象
初版では Python に対して以下に対応する。

- import 解決
- alias 解決
- スコープ解決
- qualified name 解決
- クラス / 関数 / メソッド / モジュールの識別
- `self.xxx` / `cls.xxx` の解決
- 相対 import の解決
- ブロックスコープを考慮した同名シンボルの識別
- 1 参照に対して 0 件以上の解決候補を保持すること

### 8.5 型推論の対象
初版では以下すべてを対象とする。

- 関数引数
- 戻り値
- インスタンス変数
- クラス変数
- ローカル変数

### 8.6 型未確定状態の許容
- 中間モデル生成時点では、型が不明であっても構造情報は保持する
- 型不明は `Unknown` として保持する
- 推論後も一次情報の宣言型は必要以上に破壊せず、**推論事実を別レイヤで蓄積**する
- 画面表示や図生成時に必要に応じて宣言型と推論結果を統合する
- 図生成時に型が確定できない場合は、**`Unknown` を明示表示**する

### 8.7 型推論結果の保持方針
- 型推論結果は**上書き**ではなく**事実（Fact）として 1 件ずつ蓄積**を基本とする
- 複数候補がある場合は、複数の `TypeFact` として保持し、必要に応じて `Union TypeRef` として集約可能とする
- 推論結果には、根拠となる参照・分岐条件・信頼度を保持する
- 最終表示時に `Sample1 | Sample2` のような形で集約可能とする

### 8.8 分岐時の型推論
#### 型ごとに分岐する場合
```python
return_value = func()
if isinstance(return_value, Sample1):
    ...
if isinstance(return_value, Sample2):
    ...
```
上記のようなケースでは、各分岐内での型を絞り込めるものとする。

#### 引数として渡す場合
```python
return_value = func()
self.hoge(return_value)
```
この場合、呼び出し先引数型の候補として Union を保持可能とする。

#### 関数実行する場合
```python
return_value = func()
return_value.func()
```
この場合、Union の各型ごとに到達可能な呼び出し経路を追跡し、シーケンス図では `alt` セクション等により分岐表現できる構造とする。

### 8.9 ループ時の型推論
- `for` / `while` を初版対象に含める
- ループ本体で生じる代入・呼び出し・型制約を追跡対象とする
- 初版では経路爆発を避けるため、ループは**構造として保持しつつ、追跡回数には制御を設ける**ことを許容する
- シーケンス図では `loop` セクション等により表現可能な構造とする

## 9. 起点関数要件

### 9.1 起点関数の役割
起点関数は以下の目的で用いる。

- シーケンス図の生成
- 動的型付け言語における型推論精度向上

### 9.2 起点関数指定の考え方
- **シーケンス図生成時**は起点関数の指定を必須とする
- **Python 解析時**は、クラス図やモジュール依存図のみを生成する場合でも、精度向上のための型推論機能を使うかどうかをユーザが選択できるものとする
- 起点関数はツール全体としての入口であり、シーケンス図上の途中開始点とは区別して扱う

### 9.3 起点関数の件数
- 初版では **1 件のみ**を対象とする
- 複数起点関数対応は将来対応とする

### 9.4 起点関数候補提示
ツールは起点関数候補を **GUI 上に一覧表示**し、ユーザが選択できるものとする。

#### 初版で優先的に候補とするもの
- `if __name__ == "__main__"` から到達可能な入口
- トップレベル関数
- クラスの公開メソッド相当の関数

#### 候補表示情報
最低限以下を表示する。

- モジュールパス
- クラス名
- 関数名
- 引数数
- 行番号

例:
```text
pkg.sample2.Sample2.method2(sample1, arg2)  # sample2.py:5
```

### 9.5 初版の起点関数指定方式
- 初版では **GUI の候補一覧から 1 件選択する方式のみ**を採用する
- 完全修飾名の**手入力指定は初版対象外**とし、将来対応とする
- GUI 表示時は短い表示名と詳細情報を併記し、内部的には `symbol_id` を選択結果として保持する

---

## 10. 外部ライブラリ要件

### 10.1 基本方針
- 初版では **プロジェクト内のみ解析対象**とする
- 標準ライブラリおよび 3rd パーティライブラリの内部までは解析しない

### 10.2 初版図上表現
- 初版では、**外部ライブラリは図上に非表示**とする

### 10.3 将来対応
将来的には以下の表現方法を選択可能とする。

1. note として呼び出し実績のみ補記する
2. participant として表記し、**call / return を記載**するが内部解析は行わない
3. UML に一切出力しない

### 10.4 participant 表示時の注意
- participant だけを表示し、call / return を出力しない方式は採用しない
- どこで利用されたか分からなくなり、図としての意味が薄くなるため、participant 表示を行う場合は **call / return の記載を必須**とする

---

## 11. 再帰・循環呼び出し要件

### 11.1 基本方針
- 追跡中の未 return 関数をスタック管理する
- 呼び出し対象がすでにスタック上に存在する場合、循環呼び出しまたは再帰とみなす

### 11.2 初版挙動
- 循環検知時は note を出力する
- 当該呼び出し以降の追跡は中断する

---

## 12. シーケンス図要件

### 12.1 participant 単位
- 初版では **クラス単位**で participant を表現する
- 将来的にインスタンス単位モードの追加を検討する

### 12.1.1 participant 表示名方針
- シンボル解決には完全修飾名を用いる
- 図出力では、原則として **ピリオド区切りの最終要素のみ**を participant 表示名とする
- 同名要素が衝突する場合は、**右から 1 階層ずつ表示範囲を広げる**ことで一意に識別可能な表示名を採用する
- 例:
  - `service.user.User` と `domain.user.User` が衝突しない場合 → `User`
  - `User` が衝突する場合 → `user.User`
  - なお衝突する場合 → `service.user.User` / `domain.user.User`

### 12.2 分岐表現
- `if` による条件分岐を初版対象に含める
- 型条件分岐や条件式分岐を `alt` 等で表現可能な構造とする

### 12.3 ループ表現
- `for` / `while` を初版対象に含める
- シーケンス図上では `loop` 等で表現可能な構造とする

### 12.4 追跡対象
- 起点関数からの呼び出し関係を追跡する
- 呼び出し対象がプロジェクト内で解決できる場合は追跡を継続する
- プロジェクト外である場合は 10 章の設定に従う

### 12.5 初版対象外の高度構文
初版では以下のシーケンス追跡は対象外とする。

- list 内包表記内部の詳細追跡
- 三項演算子の専用分岐表現
- `async` / `await`
- `yield`
- context manager の詳細展開

## 13. 中間モデル要件

### 13.1 設計原則
中間モデルは以下を満たすこと。

- 言語非依存であること
- 図生成に必要な情報へ抽象化されていること
- 不完全な情報を保持できること
- 言語依存情報を拡張領域へ退避できること
- Generator 側の言語分岐を最小化できること
- 推論結果を後段で蓄積・再利用できること
- 宣言情報と参照情報と推論情報を分離できること
- 動的挙動は線形結果ではなく、分岐・ループを保持可能な IR として表現できること
- 初版時点で将来拡張を見据えた骨格が確定していること

### 13.2 モデル構成
中間モデルは以下の 6 層で構成する。

#### 13.2.1 CoreModel
すべての言語で共通利用する最小要素。

- ProjectModel
- SourceUnit
- SourceSpan
- Scope
- Symbol
- SymbolReference
- ResolutionCandidate
- TypeRef
- Relation
- Diagnostic

#### 13.2.2 Semantic Payload Layer
`Symbol` の kind ごとの詳細本体を保持する層。  
`Symbol` は共通ヘッダとし、詳細情報は Payload / Decl に保持する。

- TypeDecl
- CallableDecl
- FieldDecl
- PropertyDecl
- VariableDecl
- ParameterDecl

#### 13.2.3 Behavior IR Layer
シーケンス図生成・型推論・経路追跡の一次情報となる振る舞い表現。

- BehaviorGraph
- BehaviorNode
- BehaviorEdge
- BehaviorSummary（派生要約）
- CallTraceStep（シーケンス生成時の派生結果）

#### 13.2.4 Language Extension
言語固有情報を保持するための拡張領域。

例:
- Python decorator
- Java annotation
- C# attribute
- TypeScript type alias 詳細
- C の storage class
- raw AST 由来の補助情報

#### 13.2.5 Inference Layer
推論事実や型制約を保持する。

- TypeFact
- TypeConstraint
- SymbolBindingFact

#### 13.2.6 Analysis Support Layer
解析支援のための補助情報を保持する。

- CandidateEntryPoint
- AnalysisOptionSnapshot
- UnsupportedConstructRecord

### 13.3 一次情報と派生情報の扱い
- **宣言系 Payload / Decl**を構造情報の一次情報とする
- `Relation` は宣言情報・参照情報・解決結果から組み立てる**派生ビュー / 索引情報**とする
- **BehaviorNode / BehaviorGraph**を振る舞い情報の一次情報とし、`CallTraceStep` はシーケンス図生成時の派生結果とする
- 推論結果は `TypeFact` として別保持し、宣言型を直接破壊しない

### 13.4 型参照の正規化方針
- `TypeRef` は `type_id` を持つ不変オブジェクトとして扱う
- 同一意味の型は可能な限り Type Pool により再利用する
- `Union` / `Generic` / `Array` / `Pointer` などの合成型も `TypeRef` として正規化する
- 言語固有の型表記は `raw_text` と拡張情報で保持する

### 13.5 ID 体系方針
- `project_id`、`source_unit_id`、`scope_id`、`symbol_id`、`ref_id`、`type_id`、`behavior_graph_id`、`node_id`、`diagnostic_id` など、**内部で扱う各種 ID は UUID** を用いる
- 可読性向上のための補助表示は許容するが、内部識別子の正本とはしない
- 将来拡張時も、ID の意味を名前文字列へ過度に依存しない方針を維持する

### 13.6 名称管理方針
- **シンボル解決用名称**と**図出力用表示名称**を分離する
- シンボル解決用名称は `canonical_name` として完全修飾名を保持する
- 図出力用の短縮表示名は、図種ごとの表示規則に従って派生生成する
- 表示名衝突時は、識別可能になるまで右側から順に表示範囲を広げる

## 14. データクラス構成要件

### 14.1 最適化方針
データクラスは以下を重視して構成する。

- 構造情報と推論情報を分離する
- シンボル定義と参照を分離する
- `Symbol` を共通ヘッダ、`Decl / Payload` を詳細本体として分離する
- `Relation` を正本ではなく派生ビューとして扱い、二重管理を避ける
- 型推論結果を上書きせず、事実として蓄積できるようにする
- クラス図・依存図・シーケンス図のいずれにも再利用できるようにする
- 分岐やループを含む挙動を Behavior IR で保持できるようにする
- スコープと診断情報を独立して保持できるようにする

## 15. 中間モデル生成時点の要件

### 15.1 基本方針
- 中間モデル生成時点では、型が不明でもよい
- ただし、宣言構造・参照構造・関係構造・振る舞い構造は保持する

### 15.2 例
Python では以下のような状態でも中間モデルとして成立するものとする。

```text
Class: Sample1
  Fields:
    arg1: Unknown
    sample2: Sample2

Method: method1
  Params:
    arg2: Unknown
  Return:
    Unknown

Behavior:
  IF / LOOP / CALL / RETURN は保持済み
```

### 15.3 目的
- 初期段階で構造を完成させる
- 後段の型推論で情報を上書きではなく補完する
- シーケンス図生成のために分岐・ループ情報を初期段階から失わない

## 16. 初版 Python マッピング要件

### 16.1 宣言・参照マッピング
- `class` → `TypeDecl(CLASS)`
- `def` → `CallableDecl`
- 属性代入 → `FieldDecl` または `VariableDecl`
- import → `Relation(IMPORTS)` 用の派生元情報
- 型ヒントなし → `TypeRef(UNKNOWN)`
- 継承 → `Relation(EXTENDS)` 用の派生元情報
- `@property` → `PropertyDecl`
- `@classmethod` → `CallableDecl + class method 属性`
- `@staticmethod` → `CallableDecl + static method 属性`

### 16.2 振る舞いマッピング
- 関数呼び出し → `BehaviorNode(CALL)`
- クラス生成 → `BehaviorNode(CREATE)`
- 代入 → `BehaviorNode(ASSIGN)`
- return → `BehaviorNode(RETURN)`
- `if` → `BehaviorNode(IF)`
- `for` / `while` → `BehaviorNode(LOOP)`

### 16.3 初版対象外
- list 内包表記の詳細分解
- dict / set 内包表記の詳細分解
- 三項演算子の専用ノード化
- `async` / `await`
- `yield`

### 16.4 将来対応言語のマッピング方針
以下は初版では実装しないが、将来対応時に同一中間モデルへ変換する前提とする。

#### TypeScript
- `class` → `TypeDecl(CLASS)`
- `interface` → `TypeDecl(INTERFACE)`
- `enum` → `TypeDecl(ENUM)`
- `type` → `TypeDecl(TYPE_ALIAS)` または専用拡張
- `function` → `CallableDecl`
- `A | B` → `TypeRef(UNION)`
- `T[]` → `TypeRef(ARRAY)`
- `implements` → `Relation(IMPLEMENTS)` 用の派生元情報

#### Java
- `class` → `TypeDecl(CLASS)`
- `interface` → `TypeDecl(INTERFACE)`
- `enum` → `TypeDecl(ENUM)`
- field → `FieldDecl`
- method / constructor → `CallableDecl`
- `extends` → `Relation(EXTENDS)` 用の派生元情報
- `implements` → `Relation(IMPLEMENTS)` 用の派生元情報

#### C#
- Java 相当の対応を行う
- `property` → `PropertyDecl`
- `namespace` → `SourceUnit / Symbol.canonical_name` に反映
- `partial class` → `canonical_name` 基準で統合可能な構造とする

#### JavaScript
- ES6 `class` → `TypeDecl(CLASS)`
- `function` → `CallableDecl`
- import → `Relation(IMPORTS)` 用の派生元情報
- 型不明が多いため Unknown を許容
- prototype ベースの深い解析は将来課題とする

#### C
- `struct` → `TypeDecl(STRUCT)`
- `enum` → `TypeDecl(ENUM)`
- 関数 → `CallableDecl(FUNCTION)`
- グローバル変数 → `VariableDecl`
- `#include` → `Relation(INCLUDES)` 用の派生元情報
- `char*` 等 → `TypeRef(POINTER)`
- 関数ポインタ → 将来課題とする

## 17. 非機能要件

### 17.1 拡張性
- 新規言語追加時は Parser / Resolver / Adapter を追加することで対応可能とする
- CoreModel / Semantic Payload / Behavior IR / Inference Layer の変更を最小限ではなく、**原則不要**とする
- 新規出力記法追加時は Generator の追加で対応できること
- 新規図種追加時も中間モデルの一次情報を再利用できること

### 17.2 保守性
- Core / Payload / Behavior / Extension / Inference / Analysis Support を明確に分離する
- Generator 側の言語別分岐を最小化する
- Unknown / unresolved / Union / confidence を標準的に扱えること
- 初版未対応構文は `UnsupportedConstructRecord` と `Diagnostic` により記録可能とする

### 17.3 性能
- 初期段階では単一プロジェクトの静的解析が実行可能であること
- 最適化は後続フェーズとし、初期段階では正しさ・保守性を優先する

### 17.4 障害許容
- 一部の型や依存解決に失敗しても全体解析を継続できること
- パース不能箇所があっても可能な範囲の結果を出力できること

### 17.5 診断情報レベル
- Diagnostic のレベルは **`ERROR` / `WARNING` / `INFO`** の 3 段階とする
- `ERROR` は当該単位の継続困難な解析失敗を示す
- `WARNING` は unresolved や型不明など、結果継続可能だが精度や完全性に影響する事象を示す
- `INFO` は未対応構文のスキップ記録や補足情報を示す

### 17.6 操作性
- 初版ではローカル GUI から主要操作を完結できること
- 実行時に都度変更する項目は GUI 上で直感的に指定できること
- 起点関数候補は GUI 一覧から選択可能であり、完全修飾名の手入力を要求しないこと
- PlantUML 生成結果は GUI 上で確認でき、`.puml` として保存できること

- 起点関数は初版では 1 件のみとする
- 起点関数指定は初版では GUI の候補一覧選択のみとし、手入力は将来対応とする
- 分岐・ループは初版から対象に含める
- 中間モデルは初版時点で将来拡張を見据えて確定し、将来も骨格変更しないことを原則とする
- 内部 ID は UUID とする
- シンボル解決用名称と図出力用表示名は分離する
- 型不明は `Unknown` として図上にも明示する
- Diagnostic レベルは `ERROR` / `WARNING` / `INFO` の 3 段階とする

### 19.2 不採用方針
- AST の完全コピーを中間モデルに持ち込むこと
- 初期段階から言語ごとに完全分離した中間モデルを採用すること
- 初期段階から複数起点関数対応を必須とすること
- 初期段階からインスタンス単位シーケンスを必須とすること
- 初版で Mermaid / Graphviz DOT / XMI まで同時対応すること
- 初版で外部ライブラリを図上表現すること
- participant だけを表示して call / return を出さないこと
- 初版から Web 版提供を前提とすること
- 初版で起点関数の完全修飾名手入力を必須とすること

## 21. まとめ
本ツールでは、複数言語対応を見据えつつも、初版段階では **Python のみ** を正式対応対象とする。  
ただし、中間モデルは将来の言語追加・図種追加・出力記法追加を見据え、**初版時点で骨格を確定**する。

また、初版では以下を明確なスコープとする。

- **対象言語**: Python
- **対象図**: クラス図、モジュール依存図、シーケンス図
- **出力記法**: PlantUML
- **解析の要点**: シンボル解決と型推論の分離、起点関数ベース推論、Union 許容、分岐・ループ対応、外部ライブラリ非表示
- **操作形態の要点**: ローカル GUI、実行時入力は GUI 指定、起点関数は候補一覧選択、GUI プレビュー + `.puml` 保存
- **中間モデルの要点**: Symbol / Payload 分離、Relation 派生ビュー化、TypeRef 正規化、Behavior IR 一次情報化、UUID ベース ID、解決名と表示名の分離、`Unknown` 明示

これにより、以下を両立する。

- 初版実装の現実性
- 共通化による保守性向上
- 言語追加時の拡張性
- 動的型付け言語への許容
- 図生成ロジックの再利用性
- 将来の図種追加・出力記法追加への柔軟な対応
