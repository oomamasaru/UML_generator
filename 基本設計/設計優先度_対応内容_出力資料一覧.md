# 設計優先度・対応内容・出力資料一覧

## 1. 文書目的

本書は、UML自動生成ツールの基本設計へ進むにあたり、先に整理した**設計の優先度**、各優先度で**定めるべき対応内容**、およびその結果として作成すべき**出力資料**を一覧化したものである。

本一覧は、以下の要件を前提としている。

- 初版の実装言語は Python のみ。
- 初版の解析対象言語は Python のみ。
- 初版対象図はクラス図・モジュール依存図・シーケンス図。
- 初版出力記法は PlantUML のみ。
- 単一の共通中間モデルを採用し、将来拡張時も骨格は変更しない。
- シンボル解決と型推論は別工程として扱う。
- 分岐・ループは初版から対象に含める。

## 3. 優先度ごとの詳細

### 3.1 優先度1: 設計の固定前提

#### 対応内容
- 初版で対応する対象を明確化する。
- 初版で対応しない構文・図種・出力記法を明確化する。
- 失敗しても解析継続する前提を整理する。
- 将来拡張時も変更しない設計前提を固定する。
- 「正しさ優先」「Unknown許容」「破壊的変更禁止」などの基本方針を定義する。

#### この段階で特に固定するもの
- 実装言語は Python のみ。
- 初版解析対象言語は Python のみ。
- 初版対象図はクラス図・モジュール依存図・シーケンス図。
- 初版出力記法は PlantUML のみ。
- 外部ライブラリは初版では図上非表示。
- 起点関数は初版では 1 件のみ。
- 中間モデルは初版で骨格を確定し、将来も変更しない。

#### 出力資料
- 設計方針書
- 初版スコープ一覧
- 将来対応一覧

---

### 3.2 優先度2: 中間モデル契約

#### 対応内容
- CoreModel の構成要素を定義する。
- Semantic Payload Layer の責務を定義する。
- Behavior IR Layer の正本構造を定義する。
- Inference Layer の保持対象を定義する。
- Analysis Support Layer の用途を定義する。
- ID体系、canonical_name、参照整合、Unknown の保持ルールを定義する。
- 一次情報と派生情報の境界を明確化する。

#### 先に決めるべき重点論点
- `symbol_id` / `type_id` / `scope_id` / `ref_id` / `node_id` の採番方針。
- `canonical_name` の表記ルール。
- `Relation` をいつ・どの工程で派生生成するか。
- `BehaviorGraph` を tree 主体にするか edge 主体にするか。
- `Unknown` / `unresolved` / `Union` / `confidence` の保持形式。

#### 出力資料
- データ設計書
- 中間モデル詳細設計書
- データ関連図
- ID / 命名規約定義書

---

### 3.3 優先度3: レイヤ間インターフェース

#### 対応内容
- Parser の入出力を定義する。
- Collector の入出力を定義する。
- Resolver の入出力を定義する。
- Adapter / Model Builder の入出力を定義する。
- Inferer の入出力を定義する。
- Generator の入力モデルと出力形式を定義する。
- AST を参照してよい層と、参照してはいけない層を明確化する。

#### 代表的な整理対象
- Parser: AST と構文由来補助情報。
- Collector: Symbol / SymbolReference / Scope。
- Resolver: ResolutionCandidate。
- Adapter: Payload / TypeRef / BehaviorGraph。
- Inferer: TypeFact / SymbolBindingFact。
- Generator: 中間モデルのみ参照。

#### 出力資料
- インターフェース設計書
- モジュール責務一覧
- レイヤ間データ受け渡し表

---

### 3.4 優先度4: 解析フロー詳細

#### 対応内容
- 解析全体の順序を明文化する。
- 工程ごとの開始条件・完了条件を定義する。
- 失敗時に継続する条件と中断する条件を定義する。
- 起点関数候補提示のタイミングを定義する。
- 型推論の実行条件を定義する。
- 図生成へ進める最低条件を定義する。

#### 主な観点
- 構文エラーがあっても継続可能か。
- import 未解決時の扱い。
- 型が Unknown のままでも中間モデル生成可能か。
- Diagnostic をどの工程で積むか。
- 起点関数未選択時に出力可能な図は何か。

#### 出力資料
- 処理フロー設計書
- 工程責務一覧
- 異常時フロー図

---

### 3.5 優先度5: Pythonマッピング設計

#### 対応内容
- Python AST ノードを中間モデルへどう変換するかを定義する。
- 宣言系、参照系、振る舞い系のマッピングルールを定義する。
- `self.xxx` / `cls.xxx` の解釈ルールを定義する。
- 関数呼び出しとクラス生成の判定基準を定義する。
- `@property` / `@classmethod` / `@staticmethod` の扱いを定義する。
- 初版対象外構文の記録方法を定義する。

#### 重点論点
- `self.x = ...` を FieldDecl とする条件。
- `__init__` 外の属性代入をフィールドとして扱うか。
- `Call` が `CALL` か `CREATE` かの判定ルール。
- import alias の canonical_name 正規化方法。
- list内包表記や async など未対応構文の記録方法。

#### 出力資料
- Pythonマッピング設計書
- ASTノード対応表
- 未対応構文一覧

---

### 3.6 優先度6: 図生成仕様

#### 対応内容
- クラス図の出力条件を定義する。
- モジュール依存図の出力条件を定義する。
- シーケンス図の出力条件を定義する。
- Unknown / Union / unresolved の表示方針を定義する。
- alt / loop / note の出力条件を定義する。
- 外部ライブラリ非表示時の扱いを定義する。

#### 図ごとの主な論点
- クラス図: 継承、保持、利用関係の表示条件。
- 依存図: import 単位、相対importの正規化、alias表示。
- シーケンス図: participant 名、再帰検知時の note、call / return 粒度。

#### 出力資料
- 出力仕様書
- PlantUML出力ルール一覧
- 図ごとのサンプル仕様

### 3.10 優先度10: 実装順序・マイルストーン

#### 対応内容
- 実装着手順を定義する。
- 早期に確認したい成果物を決める。
- 各段階の完了条件を定義する。
- 後回しにする難所を切り分ける。
- 最初のマイルストーンと次段階以降を整理する。

#### 推奨実装順序
1. CoreModel / Payload / Behavior / Inference の dataclass 実装。
2. 共通ユーティリティ実装。
3. Python parser。
4. symbol collector。
5. symbol resolver。
6. adapter / intermediate model builder。
7. class diagram generator。
8. module dependency generator。
9. entry point candidate extraction。
10. type inferer。
11. sequence generator。
12. diagnostic 強化。

#### 出力資料
- 実装計画書
- マイルストーン一覧
- フェーズ別完了条件一覧

## 4. 最優先で決めるべき論点

以下は、設計初期に曖昧なまま進めると後続工程への影響が大きいため、最優先で確定すべき論点である。

| 優先 | 論点 | 確定すべき内容 |
|---|---|---|
| A | canonical_name の仕様 | 比較・解決・表示の基準となる名称の統一ルール。 |
| A | ID採番ルール | symbol_id、type_id、scope_id などの一意性と再現性の方針。 |
| A | BehaviorGraph の正本構造 | tree 主体か edge 主体か、または両方保持時の整合責任。 |
| A | `self.x` の Field認定ルール | Pythonの属性代入をどう構造化するかの基準。 |
| A | Unknown / unresolved / Union の表示統合ルール | 中間モデル保持と図出力表示の接続ルール。 |

## 6. まとめ

本プロジェクトでは、まず**中間モデルとレイヤ間契約を先に固定**し、その上で解析フロー、Pythonマッピング、図生成仕様へ進む順序が最も安全である。

特に、以下の順で設計を進めるのが望ましい。

1. 設計の固定前提。
2. 中間モデル契約。
3. レイヤ間インターフェース。
4. 解析フロー詳細。
5. Pythonマッピング設計。
6. 図生成仕様。
7. 設定・操作方式。
8. エラー・診断設計。
9. テスト設計。
10. 実装順序・マイルストーン整理。

この順で進めることで、初版スコープを守りつつ、将来の多言語対応・図種追加・出力記法追加にも耐えられる基本設計を構築しやすくなる。
