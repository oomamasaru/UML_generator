# テスト設計書

## 1. 文書目的

本書は、UML自動生成ツール初版におけるテスト方針、保証範囲、観点、およびテスト種別ごとの対象を定義するものである。

---

## 2. 基本方針

### 2.1 正しさ優先

- 初版は性能最適化よりも正しさ・保守性を優先する。
- テストは「正しい構造を壊さない」「不完全情報でも壊れない」を重視する。

### 2.2 レイヤー単位保証

- Parser
- Collector
- Resolver
- Model Builder
- Inferer
- Generator

上記の各責務境界ごとに、何を保証するかを明確化する。

### 2.3 スナップショット活用

- PlantUML 出力はスナップショットまたはゴールデンテストで安定性を担保する。
- 文字列完全一致だけでなく、要点一致も併用してよい。

---

## 3. テスト分類

| 種別 | 目的 |
|---|---|
| 単体テスト | 単一モジュール・関数の責務確認 |
| 結合テスト | 複数レイヤー連携の確認 |
| スナップショットテスト | 中間モデル・PlantUML 出力の形崩れ検知 |
| ゴールデンテスト | 既知入力に対する期待成果物固定 |
| 回帰テスト | 修正で既存挙動が壊れていないこと確認 |

---

## 4. レイヤー別保証範囲

## 4.1 File Collection Layer

### 保証すること

- 対象ディレクトリ配下の Python ファイルを収集できる。
- 非 Python ファイルを除外できる。
- 収集結果を `CollectedSourceSet` へ正しく詰められる。

### 保証しないこと

- Python 構文の正しさ
- import 解決の正しさ

## 4.2 Parser Layer

### 保証すること

- Python AST を生成できる。
- class / def / import / if / for / while / return / call / create 候補を抽出できる。
- 未対応構文を `UnsupportedConstructRecord` へ記録できる。

### 保証しないこと

- symbol 解決の正しさ
- 型推論の正しさ

## 4.3 Symbol Collection Layer

### 保証すること

- `Symbol`、`Scope`、`SymbolReference` を正しく起こせる。
- owner / scope / span / usage_kind を設定できる。

## 4.4 Symbol Resolution Layer

### 保証すること

- import / alias / scope / qualified name を解決候補として保持できる。
- 1 参照に対して 0 件以上の `ResolutionCandidate` を返せる。
- 未解決時に不正確な単独決め打ちをしない。

## 4.5 Intermediate Model Generation Layer

### 保証すること

- `ProjectModel` を正規化生成できる。
- `TypeDecl` / `CallableDecl` / `FieldDecl` / `VariableDecl` / `PropertyDecl` を構築できる。
- `BehaviorGraph` / `BehaviorNode` / `BehaviorEdge` を構築できる。

## 4.6 Type Inference Layer

### 保証すること

- 宣言型を破壊せず `TypeFact` を蓄積できる。
- `isinstance` 分岐で分岐内型制約を表現できる。
- ループ追跡制御、再帰検知が働く。

## 4.7 Diagram Generator Layer

### 保証すること

- クラス図 / モジュール依存図 / シーケンス図の PlantUML を出力できる。
- Unknown / unresolved を含んでも致命的に壊れない。
- participant 名衝突を回避できる。

---

## 5. 単体テスト観点

## 5.1 Parser

- `class` を `DeclarationRecord(class)` として抽出できるか
- `def` を callable として抽出できるか
- `self.x = ...` を field candidate として抽出できるか
- `if` / `for` / `while` / `return` / `Call` を `BehaviorRecord` 化できるか
- list 内包表記を未対応構文として記録できるか

## 5.2 Collector

- class / function / parameter の `Symbol` 作成
- module / class / callable / block の `Scope` 作成
- 参照の `usage_kind` 設定

## 5.3 Resolver

- alias import 解決
- relative import 解決
- `self.xxx` 解決
- 未解決参照で candidate 0 件許容

## 5.4 Model Builder

- `FieldDecl` 生成
- `PropertyDecl` 生成
- `CallableDecl.behavior_graph_id` 接続
- `Relation(EXTENDS)` / `IMPORTS` / `CALLS` 派生

## 5.5 Inferer

- `x = Sample()` による `TypeFact`
- `self.a = x` による field 型補完
- `return x` による戻り値型補完
- `isinstance(x, A)` による branch 型制約
- 再帰 / 循環検知

## 5.6 Generator

- class のテキスト出力
- extends 矢印出力
- module import 矢印出力
- alt / loop / note 出力
- Unknown 表示

---

## 6. 結合テスト観点

## 6.1 解析基本フロー

対象:

1. ファイル収集
2. parse
3. collect
4. resolve
5. model build

保証:

- `ProjectModel` まで到達できること

## 6.2 型推論込みフロー

対象:

1. 上記基本フロー
2. 起点関数候補抽出
3. 起点選択
4. infer
5. sequence 生成

保証:

- `CallTraceStep` または等価結果を踏まえたシーケンス図が生成できること

## 6.3 失敗許容フロー

対象:

- 一部 parse 失敗
- unresolved import
- unresolved symbol
- Unknown 型

保証:

- 処理継続し、部分成果物を生成できること

---

## 7. スナップショットテスト対象

## 7.1 中間モデルスナップショット

対象:

- `ProjectModel` の要約表現
- `TypeDecl` / `CallableDecl` / `FieldDecl`
- `BehaviorGraph` のノード / edge 一覧
- `ResolutionCandidate` 一覧

目的:

- レイヤー変更で構造が崩れていないか検知する。

## 7.2 PlantUML スナップショット

対象:

- クラス図出力全文
- モジュール依存図出力全文
- シーケンス図出力全文

目的:

- レイアウトや表記崩れを検知する。

---

## 8. ゴールデンテスト対象

## 8.1 最小ケース

- class 1 件
- field 1 件
- method 1 件

## 8.2 継承ケース

- 親 class
- 子 class
- extends 1 本

## 8.3 import ケース

- 複数モジュール
- absolute import / relative import / alias import

## 8.4 シーケンス基本ケース

- 起点関数 1 件
- class 間呼び出し 2〜3 段
- return 含む

## 8.5 分岐ケース

- `if isinstance(...)`
- `alt` 出力
- branch ごとの type fact

## 8.6 ループケース

- `for` or `while`
- `loop` 出力
- 追跡回数制御

## 8.7 再帰ケース

- 再帰関数または循環呼び出し
- note 出力
- 追跡停止

## 8.8 未対応構文ケース

- list comp
- ternary
- async

保証:

- Diagnostic 記録
- 可能範囲の継続

---

## 9. 回帰テスト対象

以下の修正時は回帰対象を必須とする。

- Python parser の変更
- symbol resolution の変更
- ProjectModel dataclass の変更
- Behavior IR 変換の変更
- Type inference の変更
- PlantUML generator の変更

重点回帰観点:

- 既存のゴールデンケース差分
- Unknown / unresolved の扱い
- participant 名衝突処理
- `self.xxx` field 認定

---

## 10. テストデータ方針

### 10.1 保持形態

- 小さな Python サンプルコード群を fixtures として保持する。
- 図種別に期待結果 `.puml` を保持する。

### 10.2 粒度

- 単一構文確認用の極小 fixture
- 複数 class / module 連携 fixture
- 異常系 fixture

### 10.3 命名例

- `fixtures/parser/class_basic/`
- `fixtures/resolver/import_alias/`
- `fixtures/generator/sequence_branch/`

---

## 11. 合格基準

## 11.1 単体テスト

- 主要責務に対応するテストがすべて通ること

## 11.2 結合テスト

- 初版正式対象 3 図が最低 1 ケース以上 end-to-end で生成できること

## 11.3 ゴールデン / スナップショット

- 承認済み差分以外は発生しないこと

## 11.4 異常系

- 失敗しても継続すべきケースで継続できること
- 中断すべきケースで適切に止まること

---

## 12. 初版優先順位

### 高優先

- Parser 単体
- Resolver 単体
- ProjectModel 構築
- クラス図生成
- シーケンス図基本ケース

### 中優先

- 分岐 / ループ / 再帰ケース
- import alias / relative import
- Unknown / unresolved 表示

### 将来拡張前提

- 大規模性能試験
- 多言語共通ケース
- 出力記法差分試験

---

## 13. 決定事項まとめ

- テストはレイヤー責務ごとに保証対象を明確化する。
- Unknown / unresolved を含む正常系を必ずテストする。
- PlantUML はスナップショット / ゴールデンで安定性を担保する。
- 分岐、ループ、再帰、未対応構文は初版の重点テスト対象とする。
- 回帰テストは parser / resolver / model / inferer / generator 変更時に必須とする。
