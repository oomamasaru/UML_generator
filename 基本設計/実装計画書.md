# 実装計画書

## 1. 文書目的

本書は、UML自動生成ツール初版に向けた実装順序、マイルストーン、各段階の完了条件を定義するものである。

---

## 2. 基本方針

### 2.1 実装優先順位

初版では以下を優先する。

1. 共通中間モデルの骨格固定
2. Python 解析基盤
3. クラス図 / モジュール依存図
4. 起点関数抽出
5. 型推論
6. シーケンス図
7. 診断情報強化

### 2.2 後回し方針

以下は初版完成後または将来対応へ回す。

- Mermaid / Graphviz DOT / XMI
- PNG / SVG 出力
- 外部ライブラリ図示
- 複数起点関数
- インスタンス単位シーケンス
- 高度構文対応
- Web 版

### 2.3 完了判定方針

各フェーズは「コードがある」ではなく、

- 主要 dataclass が確定している
- テストが通る
- 最低 1 つの end-to-end ケースが動く

をもって完了とする。

## 14. リスクと対策

| リスク | 内容 | 対策 |
|---|---|---|
| モデル設計ぶれ | dataclass 実装後に責務変更が起きる | Phase 0 で設計固定を徹底 |
| `self.xxx` 解釈ぶれ | field / variable 判定揺れ | Pythonマッピング表のルールを優先 |
| CALL / CREATE 判定揺れ | `ast.Call` の判定が不安定 | resolver / adapter で確定 |
| 推論複雑化 | 分岐・ループで実装が肥大化 | Phase 6 で最小ケースから段階実装 |
| 出力差分増大 | PlantUML が頻繁に崩れる | スナップショット / ゴールデン導入 |
| GUI 先行実装による手戻り | 内部契約未確定で GUI が先走る | GUI は Phase 5 まで最小限に抑制 |

- フェーズ完了はコード存在ではなく、テストと end-to-end 動作で判定する。
- 将来対応は初版完了後に段階的に拡張する。
